Credential Redaction: Keeping Secrets Out of Logs

API keys, tokens, and passwords inevitably appear in agent conversations. Here's how q detects and redacts them before they hit disk.

AI coding agents read and write a lot of text. Some of that text contains secrets — API keys in config files, database passwords in connection strings, authentication tokens in HTTP headers. If those secrets end up in session logs, they become a security liability. q's credential redaction system catches them before they're written to disk.

The Problem

Consider a typical agent interaction:

User: "Check my config file for the database connection string"
Agent reads: config.json containing "password": "super-secret-123"
Agent responds: "Your database password is super-secret-123"

Without redaction, that password is now in the session log, potentially in the git diff, and in any exported session data. That's a credential leak.

The Redaction Pipeline

Redaction happens at multiple points in the event pipeline:

1. Event Emission

Before any event is published to the bus, it passes through the redaction filter:

; Redaction happens at event emission time
(define/contract (emit-event! event)
  (-> event? void?)
  (define redacted (redact-event event))
  (event-bus-publish! redacted))

2. JSONL Serialization

Even if an event slips through (e.g., a new event type), the JSONL serializer applies a second pass of redaction before writing to disk.

3. Tool Output

File reads and shell command outputs are redacted before the agent sees them. The agent sees [REDACTED] instead of the actual secret.

Detection Patterns

q uses pattern matching to detect common credential formats:

; Common credential patterns
(define credential-patterns
  (list
   ;; API keys (OpenAI, Anthropic, etc.)
   (regexp "sk-[a-zA-Z0-9]{20,}")
   (regexp "sk-ant-[a-zA-Z0-9]{20,}")

   ;; AWS keys
   (regexp "AKIA[0-9A-Z]{16}")
   (regexp "aws_secret_access_key")

   ;; Generic key-value patterns
   (regexp "(?i)(password|secret|token|api.key|apikey)['\"]?\\s*[:=]\\s*['\"]?([^'\"\\s,}]+)")

   ;; Bearer tokens
   (regexp "Bearer [a-zA-Z0-9._-]+")

   ;; Connection strings
   (regexp "(?i)(mongodb|postgres|mysql|redis)://[^\\s]+")))

When a match is found, the value portion is replaced with [REDACTED]:

"password": "super-secret-123"  →  "password": "[REDACTED]"
sk_placeholder_example     →  [REDACTED]
Bearer eyJhbGci...               →  Bearer [REDACTED]

The Credential Store

q maintains a runtime credential store that tracks known secrets:

  • Loaded at startup — reads API keys from config and environment variables
  • Updated dynamically — new secrets detected during sessions are added
  • Exact match — known secrets are matched exactly (not just by pattern)

This two-layer approach (patterns + exact match) catches both known and unknown secrets.

Configuration

// config.json
{
  "redaction": {
    "enabled": true,
    "mode": "strict",  // "strict" | "patterns" | "off"
    "custom-patterns": [
      "COMPANY_TOKEN_[A-Z0-9]+"
    ],
    "whitelist-fields": [
      "model"  // don't redact model names that look like keys
    ]
  }
}

What Redaction Doesn't Do

Redaction is a best-effort mitigation, not a guarantee:

  • Custom encoding (Base64, hex) may bypass pattern detection
  • Secrets split across multiple chunks might not be caught
  • The agent can still use credentials in API calls — redaction only affects logging
"Redaction is defense in depth. We also recommend never committing session logs to version control and using environment variables for all secrets." — q security documentation
