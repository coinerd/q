The Small Trusted Core Principle

ADR #1: Why q's core agent loop has minimal dependencies, no TUI or CLI coupling, and how security boundaries are enforced at the architecture level.

Our first Architecture Decision Record (ADR) established the principle that defines q's entire architecture: the agent core must be as small and dependency-free as possible. Everything else — the TUI, CLI, JSON interface, extensions — lives outside this boundary.

Why This Matters

AI coding agents have a unique threat model. They execute shell commands, read and write files, and make network requests to LLM providers. Every dependency in the core expands the attack surface and makes security auditing harder.

By keeping the core small, we achieve three things:

  1. Auditability — fewer lines of code to review for security issues
  2. Portability — the core can run anywhere without UI dependencies
  3. Stability — changes to the TUI or CLI can't break the agent loop

The Boundary

The trusted core consists of:

  • Event bus (core messaging infrastructure)
  • Event structs (typed event definitions)
  • State management (session state, turn state)
  • The main loop (iteration control, phase dispatch)
  • Tool execution (shell commands, file operations)
  • Provider abstraction (LLM interface)

Everything else — rendering, input handling, configuration, extensions — is outside this boundary.

No UI Coupling

The core has zero knowledge of the TUI. It doesn't call any rendering functions, doesn't know about terminals, and doesn't format output for display. It emits events. The TUI subscribes to those events and renders them however it wants.

This means the core can run headless, in a container, or in a test harness without any display infrastructure.

Security by Separation

Security-critical operations (shell execution, file writes) are isolated from non-critical ones (CLI argument parsing, TUI rendering). If the TUI has a vulnerability, it doesn't compromise the core.

Security is not a feature — it's the foundation. Every boundary in q's architecture is a security boundary.

Extension Points

The extension hook model (pre, post, error) allows third-party code to modify behavior without touching the core. Extensions run in their own namespace with controlled access to the event bus.
