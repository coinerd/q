Migrating to Typed Racket: Lessons Learned

A practical look at incrementally adding type annotations to a growing Racket codebase — what worked, what broke, and what we'd do differently.

Racket is dynamically typed by default. That's great for prototyping, but for a production AI coding agent that handles credentials, executes commands, and manages state machines, we wanted stronger guarantees. Typed Racket provides gradual typing — you can add types module by module. Here's how that migration went.

Why Types Matter for Agent Code

q's agent loop is critical infrastructure. A type error in the wrong place could mean:

  • Malformed prompts sent to the LLM
  • State machine transitions to invalid states
  • Session data corrupted during compaction
  • Credential leaks through improper redaction

Types don't prevent all bugs, but they eliminate entire categories at compile time.

The Boundary Approach

We didn't try to type everything at once. Instead, we created typed boundaries around the most critical modules:

; event-structs.rkt — fully typed
#lang typed/racket
(provide (struct-out turn-start-event)
         (struct-out turn-end-event)
         (struct-out tool-call-event))

(struct turn-start-event ([turn-id : String]
                          [timestamp : Exact-Nonnegative-Integer])
  #:transparent)

(struct turn-end-event ([turn-id : String]
                        [status : (U 'completed 'failed 'cancelled)])
  #:transparent)

Event structures were our first target. They're the contract between core and interfaces — getting them wrong would break everything.

What Worked

Starting with Data Structures

Struct definitions are the easiest thing to type. You know the shape of your data upfront. We typed all event structs first, then moved to configuration types, then state machine types.

Using Require/Typed for Boundaries

require/typed lets typed modules consume untyped modules with explicit type contracts:

; In a typed module
(require/typed "../core/state.rkt"
  [current-state (-> (U 'idle 'running 'paused))]
  [transition-state! (-> (U 'idle 'running 'paused) void?)])

This creates runtime boundary checks — if the untyped module returns the wrong type, you get a clear error at the boundary, not deep in your code.

Union Types for State Machines

Typed Racket's union types are perfect for state machines:

(define-type gsd-state
  (U 'exploring 'planning 'implementing
     'verifying 'debugging 'completed 'failed))

What Broke

Higher-Order Functions Are Painful

Passing functions between typed and untyped modules requires careful type specification. We had several cases where callback types didn't match exactly, causing boundary check failures at runtime.

Generic Interfaces Need Explicit Type Parameters

Racket's generic interfaces work fine untyped, but Typed Racket needs explicit type parameters. This added verbosity to provider and extension interfaces.

Compilation Time Increased

Typed Racket modules take longer to compile. Our CI went from 30 seconds to 2 minutes. We mitigated this with incremental compilation and caching.

What We'd Do Differently

  1. Type from the outside in — start with the public API contracts, not internal helpers
  2. Use : annotations everywhere — don't rely on type inference for public functions
  3. Keep the boundary thin — minimize the typed/untyped interface surface
  4. Type tests first — typed tests catch more bugs before they reach production
"Gradual typing in Racket is real. We went from 0% to 40% typed modules in two months, and the type errors we caught in CI were genuine bugs." — q contributor
