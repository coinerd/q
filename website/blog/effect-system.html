The Effect System: Typed Side Effects for Agent Operations

How q's effect system gives type-safe control over side effects — shell execution, file I/O, and network calls — without monads or async/await.

AI coding agents are fundamentally side-effect-heavy programs. They execute shell commands, read and write files, make HTTP requests, and modify global state. In most languages, these side effects are invisible — you call a function and something happens, but the type system doesn't know what. q's effect system makes side effects explicit and type-safe.

The Problem with Hidden Side Effects

Consider a typical agent operation:

(define (process-tool-call call)
  ; What does this function do?
  ; - Execute a shell command? (dangerous)
  ; - Read a file? (I/O)
  ; - Write a file? (mutation)
  ; - Make a network request? (non-deterministic)
  ; The type signature doesn't tell you.
  ...)

In a security-sensitive agent, knowing what side effects a function can perform is critical. You want to know: can this function delete files? Can it make network requests? Can it modify state?

Effect Types

q defines a set of effect types that categorize all possible side effects:

; Effect type definitions
(define-type Effect
  (U 'shell-execute    ; Execute a shell command
     'file-read        ; Read from filesystem
     'file-write        ; Write to filesystem
     'network-request   ; Make an HTTP request
     'state-read        ; Read agent state
     'state-write       ; Modify agent state
     'event-emit        ; Publish an event
     'log-write))       ; Write to session log

; Effect sets for different operation categories
(define-type tool-effects
  (Setof Effect))

(define shell-effects
  (set 'shell-execute 'state-write 'event-emit 'log-write))

(define file-read-effects
  (set 'file-read 'event-emit 'log-write))

The Effect Executor

The effect executor is the single point where side effects actually happen. Functions declare their effects; the executor validates and executes them:

; Effect executor interface
(define (execute-effect! eff)
  (match eff
    [(shell-execute cmd args)
     (unless (shell-allowed? cmd)
       (raise (effect-denied cmd)))
     (run-shell-command cmd args)]
    [(file-write path content)
     (unless (path-writable? path)
       (raise (effect-denied path)))
     (write-file path content)]
    [(network-request url headers body)
     (unless (network-allowed? url)
       (raise (effect-denied url)))
     (http-request url headers body)]))

Every side effect goes through this executor. There's no back door.

Safe Mode

Safe mode (ADR-8) restricts the allowed effects:

; Normal mode — all effects allowed
(define normal-allowed-effects
  (set 'shell-execute 'file-read 'file-write
       'network-request 'state-read 'state-write
       'event-emit 'log-write))

; Safe mode — read-only, no execution
(define safe-allowed-effects
  (set 'file-read 'state-read 'event-emit 'log-write))

In safe mode, the agent can read files and emit events, but it can't execute commands, write files, or make network requests. This is enforced at the effect executor level — even if a bug allows the agent to try an unsafe operation, the executor blocks it.

Effect Auditing

Every effect execution is logged:

{"type":"effect-executed",
 "effect":"shell-execute",
 "command":"git status",
 "allowed":true,
 "timestamp":1712345678,
 "turn-id":5}

This creates a complete audit trail of every side effect the agent performed. Combined with session replay, you can reconstruct exactly what the agent did.

Why Not Monads?

In Haskell, you'd use monads to track effects. In Racket, we have a simpler approach:

  • Functions declare their effects via contracts or type annotations
  • The executor validates at runtime
  • Safe mode restricts the allowed effect set

This is less rigorous than a monadic approach (the type system doesn't enforce effect tracking), but it's pragmatic for Racket's dynamic nature. The Typed Racket migration will add more static guarantees over time.

Benefits in Practice

  • Security — every dangerous operation passes through a single checkpoint
  • Debugging — the effect log shows exactly what the agent did
  • Testing — mock the executor to test without side effects
  • Documentation — effect declarations serve as documentation of function behavior
"The effect system is q's security backbone. It's the difference between 'the agent can do anything' and 'the agent can do exactly what we allow.'" — q contributor
