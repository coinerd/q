Session Replay: Debugging AI Agents After the Fact

How q's append-only JSONL session logs enable faithful replay, post-hoc debugging, and reproducible test cases from real usage.

Debugging an AI coding agent is fundamentally different from debugging regular software. The agent is non-deterministic — the same prompt might produce different actions each time. You can't just "re-run and see if it crashes again." That's why q logs everything to an append-only JSONL file, and why we built a replay system on top of it.

The JSONL Session Log

Every session generates a JSONL (JSON Lines) file. Each line is a typed event:

{"type":"session-start","timestamp":1712345678,"session-id":"abc123","provider":"openai","model":"gpt-4.1"}
{"type":"turn-start","timestamp":1712345680,"turn-id":1,"prompt":"Explain this function"}
{"type":"provider-request","timestamp":1712345681,"turn-id":1,"tokens-in":245,"model":"gpt-4.1"}
{"type":"stream-chunk","timestamp":1712345682,"turn-id":1,"delta":"This function"}
{"type":"stream-chunk","timestamp":1712345682,"turn-id":1,"delta":" processes the input"}
{"type":"tool-call","timestamp":1712345690,"turn-id":1,"tool":"file-read","args":{"path":"src/main.rkt"}}
{"type":"tool-result","timestamp":1712345691,"turn-id":1,"tool":"file-read","status":"ok"}
{"type":"turn-end","timestamp":1712345700,"turn-id":1,"status":"completed"}

This is an append-only log. Events are never modified or deleted. The log is the source of truth for what happened during a session.

The Replay Command

Use the q replay command to inspect past sessions:

# List all sessions
q replay list

# Replay a specific session
q replay session abc123

# Replay only tool calls
q replay session abc123 --filter tool-call

# Export to JSON for analysis
q replay session abc123 --export json > session.json

# Show a summary
q replay session abc123 --summary

Debugging Workflow

Here's how session replay helps in practice:

Step 1: Identify the Problem

A user reports that q deleted a file it shouldn't have. First, find the session:

q replay list --recent 10
# SESSION   DATE         TURNS  STATUS
# abc123    2026-04-05   23     completed
# def456    2026-04-05   8      failed

Step 2: Inspect the Events

Filter for file operations:

q replay session abc123 --filter file-write
# Turn 15: file-write src/utils.rkt (modified)
# Turn 18: file-write src/auth.rkt (modified)
# Turn 21: file-write src/deprecated.rkt (deleted) ← HERE

Turn 21 deleted a file. Let's look at the full context:

q replay session abc123 --turns 20-22 --verbose

Step 3: Understand the Agent's Reasoning

The replay shows the full context: what the agent read, what it concluded, and what prompt led to the deletion. You can see exactly why it decided to delete the file.

Extracting Test Cases

Bugs found in production become test cases. q can extract a session segment as a test:

q replay session abc123 --extract-turns 19-22 --as-test > tests/regression/deleted-file.rkt

This generates a test that replays the events and verifies the agent's behavior matches expectations.

Privacy and Redaction

Session logs may contain sensitive data. q applies credential redaction (ADR-9) to all events before writing them to the log. API keys, tokens, and passwords are replaced with [REDACTED].

The Audit Trail

Beyond debugging, session logs serve as an audit trail:

  • Compliance — prove what the agent did and didn't do
  • Analytics — measure tool usage patterns and success rates
  • Training data — successful sessions become examples for future model improvements
"Session replay turned a 'heisenbug' into a reproducible test case in 10 minutes. That alone justified the entire logging infrastructure." — q contributor
