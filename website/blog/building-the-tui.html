Building a Native TUI Without Electron

Why we chose Racket's built-in text rendering over Electron, and how the TUI subscribes to events for real-time display of streaming and goal progress.

Most AI coding tools use Electron for their UI — a web browser in disguise. We chose a different path: a native terminal user interface built directly on Racket's text rendering capabilities. Here's why, and how it works.

Why Not Electron?

Electron is the easy choice. Everyone knows HTML/CSS/JS, and the rendering is polished. But for q, it has three fundamental problems:

  • Resource overhead — 200MB+ RAM just for the UI framework. On a developer's machine already running an IDE, browser, and containers, that's significant.
  • Startup time — 2-3 seconds to boot Chromium before anything renders. q starts in under 200ms.
  • Distribution complexity — platform-specific builds, auto-update mechanisms, code signing. A Racket TUI is a single binary.

For an agent that runs in the terminal anyway, a native TUI is the natural fit.

Racket's Text Rendering

Racket provides a capable text rendering stack:

; The TUI is built on Racket's racket/text and terminal capabilities
(require racket/port
         (prefix-in term: "terminal.rkt"))

; Basic rendering
(define (render-message msg)
  (term:move-cursor row col)
  (term:write-styled msg
    #:fg (message-color msg)
    #:bold? (important? msg)))

Racket handles ANSI escape codes, cursor positioning, color, and Unicode. The TUI uses these primitives to build a rich interface.

The Event Subscription Model

The TUI doesn't poll or request data — it subscribes to the event bus:

; TUI subscribes to relevant events
(event-subscribe! bus 'stream-chunk
  (λ (event)
    (render-token (stream-chunk-delta event))))

(event-subscribe! bus 'tool-call
  (λ (event)
    (render-tool-indicator
      (tool-call-name event)
      (tool-call-args event))))

(event-subscribe! bus 'goal-transition
  (λ (event)
    (update-status-bar
      (goal-transition-from event)
      (goal-transition-to event))))

Each event type triggers a specific rendering action. The TUI is a pure consumer of events — it never modifies agent state.

The Rendering Pipeline

The TUI maintains a virtual screen buffer. When events arrive, they update the buffer. A separate render loop draws the buffer to the terminal at 60fps (or whatever the terminal supports):

; Simplified render loop
(define (render-loop)
  (sync/handle-evt
    (alarm-evt (+ (current-inexact-milliseconds) 16))  ; ~60fps
    (λ (_)
      (render-buffer-to-terminal!)
      (render-loop))))

This decoupling means events can arrive faster than the terminal can render — the buffer absorbs the difference.

Keyboard Handling

User input is handled through a dedicated input thread:

(define (input-loop)
  (define key (read-key))
  (case key
    [(enter) (handle-submit)]
    [(ctrl-c) (handle-interrupt)]
    [(ctrl-g) (handle-goal-command)]
    [(up) (history-prev)]
    [(down) (history-next)]
    [else (insert-char key)])
  (input-loop))

The input thread and render loop run concurrently. Typing doesn't block rendering, and streaming output doesn't block input.

Streaming Display

Token-by-token streaming display is the most visible feature. Each stream-chunk event appends to the current message's text, and the render loop redraws the message area:

  • Markdown is rendered in real-time (bold, code blocks, lists)
  • Syntax highlighting in code blocks
  • Word wrapping at terminal width
  • Scrolling when content exceeds the visible area

Goal Mode Visualization

When goal mode is active, the TUI shows the GSD state machine in the status bar:

┌─────────────────────────────────────────────┐
│ q agent │ Goal: exploring (3/10 turns)       │
├─────────────────────────────────────────────┤
│ ... agent output ...                        │
└─────────────────────────────────────────────┘

State transitions animate in real-time. You see the agent move from exploring to planning to implementing — it's mesmerizing.
