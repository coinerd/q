Inside the GSD State Machine: How q Executes Goals

A deep-dive into the finite-state machine that powers q's goal-directed loop — from exploration through implementation to verification.

The most ambitious feature in q is Goal Mode — give the agent a high-level objective and it autonomously plans, executes, and verifies the work. Under the hood, this is powered by the GSD (Getting Stuff Done) State Machine, an explicit finite-state machine that replaced the ad-hoc imperative logic of earlier versions.

The Problem: Ad-Hoc State Management

Before v0.25, the GSD planning system managed state through scattered flag checks and hash-table mutations. Want to know if a wave was executing? Check hash-ref(state 'executing? #f). Want to transition to verification? Set three flags and hope nothing else was reading them.

This worked — until it didn't. We found that:

  • Invalid state transitions happened silently (e.g., jumping from idle to verifying)
  • Debugging wave failures required tracing imperative code across 6 modules
  • Adding new features risked regressions in unrelated paths

The Solution: Explicit State Machine

We rewrote the executor as an explicit FSM with five states and validated transitions:

idle → exploring → plan-written → executing → verifying → idle
                                      ↑                         │
                                      └─────────────────────────┘
                                  (re-plan if verification fails)

Every transition goes through a single function (transition-state) that validates the current state, applies the transition, and emits an event. Invalid transitions raise clear errors immediately.

Canonical Runtime State

The old hash-table state was replaced with gsd-runtime-state, a struct with named fields:

(struct gsd-runtime-state
  (mode          ; 'idle | 'exploring | 'executing | 'verifying
   current-wave  ; #f or wave index
   completed     ; set of completed wave indices
   plan          ; #f or validated plan
   plan-id       ; unique plan identifier
   max-waves     ; maximum allowed waves
   created-at))  ; timestamp

All consumers use struct accessors. No more hash-ref with string keys and default values that mask bugs.

Transaction Semantics

Multi-step operations like completing a wave involve several state changes. If any step fails, we need to roll back. The with-gsd-transaction wrapper provides snapshot-and-rollback:

(with-gsd-transaction state
  (λ (s)
    (let* ([s1 (mark-wave-completed s wave-idx)]
           [s2 (update-progress s1)]
           [s3 (check-all-done s2)])
      s3)))

If any step throws, the state reverts to the snapshot. No partial mutations.

Event Telemetry

Every state transition emits an event following the gsd.<category>.<action> naming convention:

gsd.state.changed     ; state transition with from/to
gsd.plan.parsed       ; raw plan parsed from markdown
gsd.plan.normalized   ; plan converted to normalized IR
gsd.plan.validated    ; plan validated for execution
gsd.wave.started      ; wave execution begins
gsd.wave.completed    ; wave finished successfully
gsd.mode.changed      ; GSD mode switched

These events flow through the session event bus, giving the TUI, logger, and extensions real-time visibility into goal progress.

Normalized Plan IR

Plans go through a three-stage pipeline before execution:

  1. Parse — raw markdown → gsd-normalized-plan
  2. Validate — check structure, dependencies, uniqueness
  3. Execute — create wave executor from validated plan

The normalized plan is immutable — once created, it never changes. This prevents subtle bugs where the plan is modified during execution.

Results

The rewrite delivered measurable improvements:

  • 221 GSD tests covering policy, state-machine, core, planning, events, and fitness
  • Zero silent state corruption — all invalid transitions are caught immediately
  • Full observability — every transition is visible via events
  • Clean extension points — new behaviors hook into the FSM without touching core logic
