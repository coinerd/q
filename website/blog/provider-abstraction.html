The Provider Abstraction: One Agent, Many LLMs

How q's provider interface lets you swap between OpenAI, Anthropic, Gemini, and local models without changing a line of agent logic.

Most AI coding agents hard-code a single LLM provider. Want to switch from OpenAI to Anthropic? Rewrite your agent logic. Want to add a local model? Start from scratch. q takes a different approach: a provider abstraction layer that decouples agent logic from model specifics.

The Problem

LLM providers differ in fundamental ways:

  • API shapes — different request/response formats
  • Streaming protocols — SSE vs. WebSocket vs. custom
  • Token counting — each provider counts differently
  • Tool calling — function calling formats vary wildly
  • Rate limits — different quotas and retry strategies

Without abstraction, every provider change touches the agent core. That violates the small trusted core principle (ADR-1).

The Provider Interface

q defines a minimal provider contract that all LLM backends implement:

; The provider interface
(provide gen:provider)
(define gen:provider
  (interface ()
    [complete  (-> prompt-state? (values response? provider-state?))]
    [stream    (-> prompt-state? (sequenceof chunk?))]
    [count-tokens (-> string? exact-nonnegative-integer?)]
    [models    (-> (listof model-info?))]))

Four methods. That's it. Every provider — OpenAI, Anthropic, Gemini, Ollama — implements these four operations. The agent core never sees provider-specific details.

Streaming Across Providers

Streaming is the trickiest part. Each provider sends chunks differently:

  • OpenAI — SSE with data: prefixed lines
  • Anthropic — SSE with event type headers
  • Local models — often newline-delimited JSON

q normalizes all streaming into a uniform (sequenceof chunk?) — Racket's lazy sequences handle backpressure naturally. The provider yields chunks as they arrive; the consumer pulls at its own pace.

Configuration

Switching providers is a config change, not a code change:

// config.json
{
  "provider": "anthropic",
  "model": "claude-sonnet-4-20250514",
  "api-key": "${ANTHROPIC_API_KEY}"
}

Or use the CLI flag:

q --provider openai --model gpt-4.1 chat

Adding a New Provider

Want to integrate a new LLM? Implement the four-method interface:

  1. Create a new module in providers/
  2. Implement complete, stream, count-tokens, models
  3. Register the provider in the provider registry
  4. Add your API key to config

No changes to the agent loop, the event bus, or any interface code.

Fallbacks and Routing

q supports provider fallbacks — if the primary provider fails, it automatically tries the next:

{
  "providers": [
    {"name": "openai", "model": "gpt-4.1"},
    {"name": "anthropic", "model": "claude-sonnet-4-20250514"}
  ],
  "fallback": true
}

Rate limit hit on OpenAI? q transparently switches to Anthropic for that request. The agent doesn't know or care.
