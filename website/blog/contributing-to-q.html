Contributing to q: A Practical Guide

Everything you need to set up your dev environment, write tests, and submit your first PR to the q agent project.

q is open source and we love contributions. Whether you're fixing a bug, adding a feature, improving documentation, or writing tests, here's everything you need to get started.

Setting Up Your Development Environment

# Clone the repository
git clone https://github.com/coinerd/q.git
cd q

# Install Racket (if you don't have it)
# macOS:
brew install --cask racket
# Linux:
sudo apt install racket  # or your package manager

# Install dependencies
raco pkg install

# Verify everything works
raco test .

Project Structure

├── main.rkt              # Entry point
├── core/
│   ├── event-bus.rkt     # Central event bus
│   ├── event-structs.rkt # Event type definitions
│   ├── loop.rkt          # Main agent loop
│   ├── state.rkt         # Session state management
│   └── ...
├── runtime/
│   ├── context-policy.rkt  # Shared context policy
│   ├── context-manager.rkt # Production context pipeline
│   ├── loop-config.rkt     # Agent loop configuration
│   └── ...
├── extensions/
│   ├── gsd/              # Goal mode state machine
│   └── ...
├── tui/                  # Terminal UI
├── cli/                  # CLI interface
├── json/                 # JSON streaming mode
├── tools/                # Built-in tools
├── providers/            # LLM provider implementations
├── docs/                 # Documentation & ADRs
├── website/              # This website!
└── tests/                # Test suite

Running Tests

# Run all tests
raco test .

# Run a specific test file
raco test tests/event-bus-test.rkt

# Run only GSD tests
raco test extensions/gsd/

# Run with verbose output
raco test -v .

Code Style

  • Follow Racket conventions: kebab-case for identifiers
  • Use provide and require explicitly — no implicit imports
  • Document public functions with docstrings
  • Keep functions short — if it's over 30 lines, consider splitting
  • Add type annotations in Typed Racket modules

The PR Process

  1. Fork the repository
  2. Create a branch: git checkout -b my-feature
  3. Write code and tests
  4. Run the test suite: raco test .
  5. Commit with a clear message
  6. Open a PR against main

Writing Good Tests

q uses RackUnit. Here's a pattern for event bus tests:

(require rackunit
     "../../core/event-bus.rkt")

(define bus (make-event-bus))
(define received '())

(event-bus-subscribe bus 'test-event
  (λ (data) (set! received (cons data received))))

(event-bus-emit! bus 'test-event "hello")

(check-equal? received '("hello"))

Architecture Decision Records

For significant changes, we write ADRs (Architecture Decision Records). These live in docs/adr/. An ADR has three sections:

  1. Context — what's the situation?
  2. Decision — what did we choose?
  3. Consequences — what are the trade-offs?

Even if you're not writing a formal ADR, thinking in these terms helps write better PR descriptions.

Getting Help

  • GitHub Issues — for bugs and feature requests
  • GitHub Discussions — for questions and design ideas
  • CODEOWNERS — check who to tag for review

We're friendly and responsive. Don't hesitate to open a draft PR early for feedback!
