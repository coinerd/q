Context Compaction: When Conversations Get Too Long

Long sessions blow past token budgets. Here's how q detects context pressure, summarizes old messages with LLM assistance, and keeps conversations coherent.

LLMs have token limits. A long coding session — exploring code, reading files, fixing bugs — generates thousands of tokens of conversation history. Eventually, the context window fills up. If you don't handle this, the agent either fails (provider error) or degrades (truncated context, lost instructions). q's context compaction system solves this.

Token Pressure Detection

q monitors token usage in real-time. After each turn, it estimates the total context size:

; Context pressure events
(struct context-pressure-event
  ([current-tokens : Exact-Nonnegative-Integer]
   [max-tokens : Exact-Nonnegative-Integer]
   [pressure-level : (U 'low 'medium 'high 'critical)]))

Pressure levels:

  • Low (< 50%) — plenty of room
  • Medium (50-70%) — start planning compaction
  • High (70-85%) — compaction is needed soon
  • Critical (> 85%) — compaction must happen before next turn

The Compaction Strategy

When pressure reaches high or critical, q compacts the conversation. The strategy is simple but effective:

1. Preserve System Instructions

The system prompt and safety instructions are never compacted. They always remain at full fidelity.

2. Keep Recent Turns

The last N turns are preserved verbatim. Recent context is most relevant to the current task.

3. Summarize Old Turns

Older turns are replaced with an LLM-generated summary:

; Compaction prompt (simplified)
(define compaction-prompt
  "Summarize the following conversation turns, preserving:
   - Key decisions made
   - Files read or modified
   - Errors encountered and how they were resolved
   - Important conclusions about the codebase
   Discard small talk and repetition.")

The LLM reads the old turns and produces a compact summary that captures the essential information.

4. Preserve Tool Results

File contents and tool outputs that were recently referenced are kept. Tool outputs that haven't been referenced in N turns are summarized.

The Compaction Event

Compaction is transparent but observable. q emits a context-compaction event:

{"type":"context-compaction",
 "timestamp":1712345678,
 "turns-before":24,
 "turns-after":8,
 "tokens-before":78432,
 "tokens-after":31200,
 "summarized-turns":"1-16",
 "preserved-turns":"17-24"}

This event is logged in the session log, so you can always see when and how compaction happened.

Preserving Important Context

Not all turns are equal. q prioritizes keeping:

  • Goal descriptions — the current objective is always preserved
  • Code diffs — recent file modifications
  • Error messages — they often contain the key to a solution
  • Conclusions — recorded conclusions from the goal loop

The Honest Trade-off

Compaction is lossy. Information is discarded. The summary captures the main points, but details fade. This is why q records conclusions during goal execution — they serve as a more durable memory than the raw conversation.

In practice, we've found that:

  • Sessions under 20 turns rarely need compaction
  • Goal mode sessions (which are longer) typically need 1-2 compactions
  • The summary quality is usually sufficient for the agent to continue productively
  • When compaction causes issues, it's visible in the session replay

Configuration

// config.json
{
  "context": {
    "compaction-threshold": 0.75,
    "preserve-recent-turns": 6,
    "summarization-model": null  // null = use same model
  }
}

You can use a cheaper model for summarization — it doesn't need to be the same model that's doing the coding.
