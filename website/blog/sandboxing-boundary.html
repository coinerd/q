Sandboxing LLM-Generated Shell Commands

Security is not optional. Here's how q uses shell quoting, resource limits, and timeout-based sandboxing to safely execute AI-generated commands.

AI coding agents can execute shell commands. That's what makes them powerful — and what makes them dangerous. A misinterpreted prompt or a confused model could run rm -rf / or exfiltrate environment variables. Security isn't optional; it's foundational.

The Threat Model

We design against three threat categories:

  1. Prompt injection — malicious instructions hidden in files the agent reads
  2. Model confusion — the LLM generates destructive commands by mistake
  3. Tool escape — a compromised tool tries to access the host system

We can't prevent all of these — but we can reduce the blast radius.

Layer 1: Shell Quoting

All shell commands pass through strict quoting before execution:

; Commands are constructed with explicit argument lists
(shell-exec "git" '("commit" "-m" "fix: resolve timeout issue"))

; Never: (shell-exec (format "git commit -m ~a" user-input))

Arguments are individually quoted, preventing shell expansion and injection.

Layer 2: Subprocess Limits

Shell commands run through a managed subprocess with configurable restrictions:

  • Timeouts — commands that take too long are killed
  • Working directory — commands run in the project root, not /
  • Allow/deny lists — optionally restrict which commands can run
  • Resource limits — CPU and memory caps on subprocesses

Layer 3: File Scope

File tools operate within a configurable project root:

; File reads are scoped to the project directory
(file-read "src/main.rkt")  ; ✅ within project
(file-read "/etc/passwd")   ; ❌ path traversal blocked

Path traversal attacks (../../etc/passwd) are caught and rejected.

Layer 4: Racket Sandbox

For code evaluation, q uses Racket's built-in sandbox:

  • Restricted module imports
  • Memory and CPU limits
  • No filesystem access from evaluated code
  • No network access

Layer 5: Safe Mode

For maximum protection, enable safe mode:

q --safe-mode chat

Safe mode adds confirmation prompts for all destructive operations. Every file write, shell command, and network request requires explicit user approval.

The Honest Assessment

Sandbox escape is still possible with sufficient effort — this is a mitigation, not a guarantee.

We're honest about the limits. These layers reduce risk significantly, but a determined attacker with prompt injection could potentially bypass them. That's why we also emphasize:

  • Audit logs — every command is recorded in the session log
  • Replay — you can replay sessions to see exactly what ran
  • Safe mode by default — for new users, safe mode is on
