Context Assembly: Building the Right Prompt at the Right Time

How q dynamically assembles context for each turn — combining system prompts, conversation history, tool results, and extension hooks.

Every turn, q sends a prompt to the LLM. That prompt isn't just the user's message — it's a carefully assembled context that includes system instructions, conversation history, tool definitions, extension hooks, and budget constraints. Getting this right is critical for both quality and cost.

The Context Assembly Problem

LLMs have context windows — maximum token limits per request. GPT-4o supports 128K tokens, Claude supports 200K. But fitting a long conversation with tool calls, file contents, and extension outputs into that budget requires strategy.

The naive approach — just concatenate everything — fails because:

  • Long conversations exceed the budget
  • Old messages may be less relevant than recent ones
  • Tool results can be very large (e.g., file contents)
  • Extension hooks add unpredictable context

The Three-Module Architecture

We unified context assembly into three modules with clear responsibilities:

1. context-policy.rkt — Shared Policy

The source of truth for:

  • estimate-message-tokens — canonical token estimation
  • ensure-first-user-pinned — first user message always included
  • fit-messages-pair-preserving — budget fitting that keeps user/assistant pairs together

Both the production pipeline and the fallback import from this single source. No more divergent implementations.

2. context-manager.rkt — Production Pipeline

The primary context assembly pipeline with six stages:

assemble-context:
  1. Tree-walk     → enumerate all context entries
  2. Pin           → ensure first user message is included
  3. Budget        → fit messages within token budget
  4. Summarize     → LLM-summarize excluded entries
  5. Catalog       → generate observability metadata
  6. Reassemble    → build final message list

When a provider is available, excluded entries are LLM-summarized instead of dropped. This preserves context quality while staying within budget.

3. context-builder.rkt — Legacy Fallback

Used when no provider is configured (e.g., offline mode). Does the tree-walk and budget fitting without summarization.

Pair-Preserving Budget Fitting

A key insight: user messages and assistant responses form pairs. Dropping one without the other creates confusing context. Our fit-messages-pair-preserving algorithm:

  1. Starts with the most recent messages
  2. Adds messages backward, keeping user/assistant pairs together
  3. Always includes the first user message (pinned)
  4. Stops when the budget is reached

Summary Caching

LLM summarization is expensive — it requires an API call. We cache summaries for excluded entries so that subsequent turns don't re-summarize the same content. The cache is invalidated when the conversation structure changes.

Observability

The catalog stage generates metadata about what was included, excluded, and summarized. This is emitted as an event, giving the TUI and extensions visibility into context assembly decisions.
