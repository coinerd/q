Streaming Architecture: Token-by-Token to Screen

The journey of a single token: from the provider's SSE stream through q's event bus to the TUI renderer — with backpressure and error handling.

When you see text streaming onto your screen in q's TUI, there's a lot happening behind the scenes. A single token travels from the LLM provider's server, through network connections, into Racket's event system, and finally onto your terminal. Here's how that pipeline works.

The Journey of a Token

  1. Provider SSE — the LLM sends a Server-Sent Event with a text delta
  2. HTTP Client — Racket's HTTP client receives the SSE chunk
  3. Provider Adapter — normalizes the provider-specific format into a generic chunk
  4. Stream Runner — manages the active stream lifecycle
  5. Event Bus — publishes a stream-chunk event
  6. JSONL Logger — appends the event to the session log
  7. TUI Renderer — subscribes to the event and draws the character

SSE Parsing

Each provider sends SSE differently. OpenAI uses data: {json} lines, Anthropic uses event-type headers. The provider adapter normalizes this:

; Simplified SSE chunk parsing
(define (parse-sse-chunk raw-bytes)
  (define line (bytes->string/utf-8 raw-bytes))
  (cond
    [(string-prefix? line "data: ")
     (define json-str (substring line 6))
     (define data (string->jsexpr json-str))
     (extract-delta data)]
    [(string=? line "")
     #f]  ; keep-alive newline
    [else #f]))

The parser runs in a dedicated thread, reading from the HTTP connection and yielding chunks to the stream runner.

The Stream Runner

The stream runner manages active streams — starting, pausing, and cancelling them:

(struct active-stream
  ([id : String]
   [thread : Thread]
   [port : Input-Port]
   [status : (U 'running 'paused 'completed 'error)]
   [buffer : (Listof String)]))

Each stream gets its own thread. The thread reads from the HTTP port, parses chunks, and publishes events. The main agent loop can check stream status without blocking.

Backpressure

What happens when the LLM generates tokens faster than the TUI can render them? Racket's lazy sequences handle this naturally:

  • The stream thread produces chunks into a bounded channel
  • The event bus reads from the channel at its own pace
  • If the channel fills up, the stream thread blocks until there's room
  • The TUI subscribes asynchronously — slow rendering doesn't block the stream

This means the TUI might lag slightly behind, but it never causes the stream to lose data.

Error Recovery

Network connections fail. Providers return errors. q handles streaming errors gracefully:

  1. Connection lost — the stream runner detects the closed port and publishes a stream-error event
  2. Provider rate limit — the adapter catches the 429 response and triggers the retry hook
  3. Partial response — accumulated chunks are preserved; the partial response is still usable
  4. Timeout — streams that idle too long are cancelled automatically

In all cases, the partial response is preserved. You never lose what was already generated.

The Streaming Port Lifecycle

Streaming ports (ADR-10) follow a strict lifecycle:

closed → opening → open → closing → closed
                    ↓         ↑
                   draining

Each transition is validated. You can't write to a closed port. You can't close a port that's already closing. This prevents resource leaks and race conditions.

Performance

On a typical streaming response, the end-to-end latency from provider to screen is under 5ms. The bottleneck is always the network, not q's pipeline. The event bus handles 10,000+ events per second — far more than any LLM generates.

"Streaming is one of those things that seems simple until you actually build it. SSE parsing, backpressure, error recovery, and lifecycle management each have subtle edge cases." — q contributor
