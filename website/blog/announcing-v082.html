Announcing q v0.82 — Goal-Directed Autonomous Loops

The biggest feature since session branching. Give q an objective and it autonomously plans, executes, and verifies until done — with full TUI integration, evidence-driven behavior, and configurable evaluators.

Today we're releasing q v0.82, which introduces the goal loop — q's new autonomous execution engine. This is the culmination of months of work on the GSD (Getting Stuff Done) state machine and represents a fundamental shift in how you interact with the agent.

What is Goal Mode?

Instead of sending individual prompts and waiting for responses, you declare a high-level goal:

/goal fix all failing tests in the auth module

q then autonomously drives through a state machine: exploring → planning → implementing → verifying → debugging. It reads files, understands the codebase, writes a plan, makes changes, runs tests, and fixes failures — all without further input from you.

"The goal loop is like having a junior developer who never gets tired, always follows the plan, and shows their work." — early tester

Evidence-Driven Behavior

Unlike typical agent loops that stop when they "feel done," q's goal loop is evidence-driven. At each turn, the system injects context about:

  • Turn count and remaining budget
  • The original goal text
  • Instructions for what constitutes verifiable success

This means q doesn't just claim it fixed the tests — it runs them, captures the output, and reports the results.

TUI Integration

Goal mode is fully integrated into the TUI. When a goal is active, you can see the state machine transitions in real time, watch the agent explore files and write plans, and interrupt at any point with /goal clear.

Configurable Evaluators

Different tasks need different success criteria. q supports multiple evaluator profiles:

  • default — standard evidence collection
  • strict — requires passing tests, clean builds, no regressions
  • relaxed — accepts partial progress
/goal --evaluator strict make all tests pass

Technical Details

Under the hood, goal mode runs in a dedicated thread within the current session. The GSD state machine (ADR-11) validates every state transition — invalid transitions raise clear errors instead of silently corrupting state. A unified policy module (extensions/gsd/policy.rkt) centralizes all guard decisions.

What's Next

This is just the beginning. Future releases will add:

  • Multi-goal orchestration (breaking large goals into sub-goals)
  • Goal templates for common workflows
  • Visual goal progress in the TUI
  • Goal history and replay

Try it today: update to v0.82 and run /goal in the TUI. We'd love to hear your feedback on GitHub Discussions.
