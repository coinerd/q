The Extension Hook Model: Pre, Post, and Error

How q's hook system lets extensions intercept and modify agent behavior at critical points — without touching core code.

Extensions are how q grows without becoming a monolith. But for extensions to be useful, they need to intercept agent behavior at the right moments. That's where hooks come in — a three-phase interception model that gives extensions control without compromising the core.

Why Hooks?

Without hooks, extending an agent means either:

  • Modifying core code (fragile, risky)
  • Wrapping the entire agent (too coarse-grained)
  • Using callbacks everywhere (callback hell)

Hooks provide structured interception points. Extensions register functions at specific lifecycle moments. The core calls them automatically.

The Three Hook Types

Pre-hooks: Before It Happens

Pre-hooks run before an action. They can inspect, modify, or cancel the action:

; Register a pre-hook for shell commands
(define-hook 'pre-shell-exec
  (λ (cmd args)
    (when (member (car args) '("rm" "rmdir" "format"))
      (log-warning "Blocked destructive command: ~a" cmd)
      (hook-cancel "Destructive command blocked"))))

Pre-hooks receive the action's arguments and can return:

  • Modified arguments — change what the action sees
  • Cancel — prevent the action entirely
  • Pass-through — let the action proceed unchanged

Post-hooks: After It Happens

Post-hooks run after an action completes. They receive the result and can transform it:

; Log all file writes for audit
(define-hook 'post-file-write
  (λ (path content result)
    (audit-log! "file-write" path)
    result))  ; return the original result

Post-hooks are perfect for observability — logging, metrics, and notifications.

Error-hooks: When Things Go Wrong

Error hooks fire when an action fails. They can retry, fallback, or transform the error:

; Retry failed provider calls with exponential backoff
(define-hook 'error-provider-call
  (λ (error context)
    (if (rate-limit-error? error)
        (begin
          (sleep (expt 2 (context-retry-count context)))
          (hook-retry))
        (hook-propagate error))))

Hook Registration

Hooks are registered in extension initialization:

#lang racket
(provide (all-defined-out))
(require q/extension)

(define-extension audit-extension
  #:name "audit-logger"
  #:hooks
  (list (pre-hook 'file-read #'audit-file-read)
        (pre-hook 'file-write #'audit-file-write)
        (post-hook 'shell-exec #'log-command-result)
        (error-hook 'provider-call #'handle-provider-error)))

Extensions declare their hooks upfront. The core validates them at load time — invalid hook points are caught immediately, not at runtime.

Execution Order

Multiple extensions can register hooks at the same point. Execution order is deterministic:

  1. Pre-hooks run in registration order
  2. The core action executes
  3. Post-hooks run in registration order
  4. If any step fails, error hooks run in reverse order

This predictability makes debugging extensions straightforward.

Available Hook Points

q exposes hook points across the agent lifecycle:

  • turn-start / turn-end — agent turn boundaries
  • file-read / file-write — file operations
  • shell-exec — shell command execution
  • provider-call — LLM provider calls
  • goal-transition — GSD state machine transitions
  • context-assembly — prompt construction
