Why We Built an Event Bus for an AI Agent

How a deterministic event bus decouples q's agent loop from its interfaces, enabling real-time streaming, metrics, and audit logging without tight coupling.

When you build an AI coding agent, you quickly realize that everything wants to know what's happening: the TUI needs to update, the logger needs to persist, extensions want to react, and the JSON interface needs to stream. The naive approach — calling each consumer directly — creates a brittle web of dependencies.

The Coupling Problem

Imagine the agent loop directly calling the TUI renderer after each token:

; ❌ Bad: every consumer must be called explicitly
(define (process-token token)
  (tui-render token)
  (logger-write token)
  (json-stream token)
  (extension-notify token)
  ; ... add every new consumer here

Every new feature that needs token data requires modifying the core loop. That violates our core principle: the agent core should be minimal.

Enter the Event Bus

The event bus is q's central nervous system. Instead of the core calling consumers directly, it emits events:

; ✅ Good: core only emits, doesn't know about consumers
(define (process-token token)
  (event-bus-emit! bus 'token-received token))

Consumers subscribe independently:

; TUI subscribes
(event-bus-subscribe bus 'token-received render-token)

; Logger subscribes
(event-bus-subscribe bus 'token-received log-token)

; Extension subscribes
(event-bus-subscribe bus 'token-received on-token)

The core doesn't know or care who's listening. New consumers are added without touching production code.

Typed Events

Events aren't just strings — they're typed structs with specific fields. This gives us:

  • Compile-time safety (in Typed Racket modules)
  • Documentation — each event type has defined fields
  • Pattern matching — subscribers can match on event shape

Our event taxonomy covers 9 categories: session, message, tool, stream, turn, provider, hook, iteration, and context-pressure events.

Deterministic Dispatch

Events are dispatched synchronously in subscription order within a single agent turn. Why not async?

  1. Testability — the order of side effects is deterministic
  2. Replay — recorded events replay identically
  3. Debuggability — you always know which subscriber runs next

Async boundaries exist at the turn level (between LLM calls), not within a turn.

What Flows Through the Bus

Session events:    session.started, session.ending, session.paused
Message events:    user.message, assistant.message, system.message
Tool events:       tool.invoked, tool.result, tool.error
Stream events:     stream.token, stream.started, stream.completed
Turn events:       turn.started, turn.completed, context.assembled
Provider events:   provider.call.started, provider.response, provider.error
Hook events:       hook.pre, hook.post, hook.error
Iteration events:  loop.iteration, goal.progress
Context pressure:  context.budget.warning, context.compaction

Real-World Impact

The event bus has enabled features we couldn't have built with direct coupling:

  • Session replay — subscribe to recorded events and replay exactly what happened
  • Metrics dashboard — a subscriber that counts events and reports throughput
  • Extension hooks — extensions subscribe to pre/post events without modifying core
  • JSON streaming — a pure event consumer that outputs JSON lines
