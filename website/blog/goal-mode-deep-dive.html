Getting Started with Goal Mode

Step-by-step guide to using q's autonomous goal loop. Set objectives, watch q plan and execute, and review the evidence trail.

Most AI coding tools are chat-based: you ask, they answer. q goes further with Goal Mode — give it an objective and it autonomously plans, executes, and verifies until the job is done. Here's how to get the most out of it.

Setting Your First Goal

# From the command line
q goal "Add input validation to the registration form"

# From within the TUI
/goal Add input validation to the registration form

That's it. q takes the goal and starts working.

What Happens Next

The GSD state machine kicks in with five phases:

1. Exploration

q reads your codebase to understand the relevant files. It looks at the registration form, related models, existing validation patterns, and test files. You'll see file reads in the TUI as it explores.

2. Planning

q creates a structured plan with specific phases:

## Plan: Add input validation to registration form

### Phase 1: Analysis
- Identify all form fields requiring validation
- Review existing validation utilities

### Phase 2: Implementation
- Add email format validation
- Add password strength validation
- Add required field validation

### Phase 3: Testing
- Write unit tests for each validator
- Run existing tests to verify no regressions

3. Implementation

q executes each phase of the plan. It writes code, edits files, and runs tools — all visible in real-time through the TUI.

4. Verification

q runs tests and checks results. If tests fail, it transitions to debugging mode, fixes the issue, and re-verifies.

5. Completion

q reports the results with evidence: tests passed, files modified, and a summary of changes.

Specifying Acceptance Criteria

Make goals more precise with criteria:

q goal --criteria "All tests pass" \
       --criteria "No lint warnings" \
       --criteria "Coverage ≥ 80%" \
       "Refactor the authentication module"

Tips for Better Goals

  • Be specific — "Add email validation to the registration form" works better than "Make the form better"
  • Set boundaries — use --max-turns to limit depth: q goal --max-turns 5 "Fix the timeout bug"
  • Review the plan — pause after the planning phase to verify q's approach before it starts writing code
  • Start small — break large goals into smaller, well-scoped ones

Monitoring Progress

In the TUI, you'll see:

  • State transitions (exploring → planning → implementing → verifying)
  • File reads and writes in real-time
  • Test results as they come in
  • Conclusions q records about what it learned

The session log captures everything, so you can always replay and review later.
