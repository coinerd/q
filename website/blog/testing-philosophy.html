Testing Philosophy: 221 Tests and Counting

How q achieved comprehensive test coverage for event bus, state machines, and provider integrations — with strategies for testing non-deterministic LLM outputs.

Testing an AI coding agent is fundamentally different from testing regular software. The core of the system — the LLM — is non-deterministic. The same prompt can produce different responses. Tool calls depend on model reasoning. State machine transitions depend on model interpretation of results. Here's how we built 221 tests (and counting) for a fundamentally non-deterministic system.

Test Categories

q's tests fall into four categories:

Unit Tests (140+)

Pure function tests with no external dependencies. These cover:

  • Event struct construction and predicates
  • JSON serialization/deserialization
  • Credential redaction regex patterns
  • Configuration parsing and validation
  • Queue and stream data structures
; Example: Event predicate test
(test-case "turn-start-event predicate"
  (define evt (turn-start-event "t1" 12345))
  (check-true (turn-start-event? evt))
  (check-equal? (turn-start-event-turn-id evt) "t1")
  (check-false (turn-end-event? evt)))

Integration Tests (50+)

Tests that exercise multiple components together:

  • Event bus publish/subscribe cycles
  • Stream runner lifecycle (start, pause, resume, cancel)
  • Tool turn bridge (tool call → execution → result)
  • JSONL serialization round-trip

State Machine Tests (20+)

The GSD state machine is the most complex component. We test every valid and invalid transition:

(test-case "GSD: valid exploration → planning transition"
  (define fsm (make-gsd-fsm))
  (gsd-transition! fsm 'begin-plan)
  (check-equal? (gsd-current-state fsm) 'planning))

(test-case "GSD: invalid exploration → completed transition"
  (define fsm (make-gsd-fsm))
  (check-exn exn:fail?
    (λ () (gsd-transition! fsm 'task-complete))))

Provider Mock Tests (11+)

Provider tests use mock HTTP responses to avoid actual API calls:

; Mock provider for testing
(define mock-provider
  (make-mock-provider
    #:responses
    (list (mock-response
            #:text "I'll read the file for you."
            #:tool-calls (list (mock-tool-call "file-read" '(("path" . "src/main.rkt"))))))))

Testing Non-Deterministic LLM Outputs

You can't assert exact text from an LLM. Instead, we test structural properties:

  • Response parsing — does the parser handle valid JSON responses?
  • Tool call extraction — can we extract tool calls from a model response?
  • Streaming assembly — do chunks assemble into a complete response?
  • Error handling — does the agent recover from malformed responses?

The Mock Provider

Our mock provider is the key to deterministic testing:

(define (make-mock-provider #:responses responses)
  (λ (prompt)
    (define resp (car responses))
    (set! responses (cdr responses))
    (values (mock-response-text resp)
            (mock-response-tool-calls resp))))

It returns pre-defined responses in sequence. Tests become deterministic and repeatable.

Test Conventions

We follow a few conventions documented in TEST_CONVENTIONS.md:

  • Tests run in CI with raco test
  • Fast tests (< 30s total) are separated from slow integration tests
  • Architecture tests verify ADR compliance
  • TUI tests use a virtual terminal (no real terminal needed)

Coverage Gaps

We're honest about what's not tested:

  • Multi-turn agent behavior — hard to mock the full reasoning chain
  • Error recovery in production — requires real network conditions
  • TUI visual correctness — we test logic, not pixel-perfect rendering

These gaps are covered by manual testing and the session replay system.

"Testing an AI agent is like testing a co-pilot. You can't predict what they'll say, but you can verify they follow the right procedures." — q contributor
