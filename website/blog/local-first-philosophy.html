Why Local-First Matters for AI Coding Agents

Your code stays on your machine. Your API keys stay on your machine. Here's why we designed q to be local-first and what that means in practice.

Many AI coding tools require uploading your codebase to a cloud service. Your source code, configuration files, and sometimes even credentials travel to a server you don't control. q was designed differently — it runs on your machine, and your code never leaves it.

What Local-First Means

Local-first doesn't mean "no internet." q still calls LLM APIs (OpenAI, Anthropic, etc.) for inference. Local-first means:

  • Code never leaves your machine — file reads happen locally, not through a cloud proxy
  • API keys stay on your device — stored in your local config, not uploaded to a service
  • Session logs are local files — not uploaded to anyone's cloud
  • Commands execute locally — shell commands run on your machine, not a remote VM
  • No account required — install, configure your API key, and go

The Privacy Argument

Your source code is intellectual property. It contains business logic, algorithms, and trade secrets. Uploading it to a third-party service creates risk:

  • Data breaches at the service provider
  • Terms of service that grant training rights on your code
  • Government subpoenas accessing your code
  • Employee access at the service provider

With q, only the LLM API calls leave your machine — and those contain only the prompts and context you choose to send, with credential redaction applied.

The Offline Capability

While q needs network for LLM inference, many features work offline:

# These work without internet
q replay list          # Browse session history
q replay session abc   # Replay a past session
q inspect              # Inspect local config and state

# These need network
q chat                 # Needs LLM API
q goal "..."           # Needs LLM API

For full offline capability, pair q with a local model via Ollama or llama.cpp. The provider abstraction makes this seamless.

The Trust Model

Local-first creates a clear trust boundary:

  • You trust: your machine, your LLM provider (for inference only)
  • You don't need to trust: q's servers (there are none), a cloud hosting service, a SaaS provider

The only network calls q makes are to the LLM provider you configure. We publish a dependency policy that documents every network dependency.

No Telemetry

q does not collect telemetry. No usage metrics, no crash reports, no feature usage tracking. The only data that leaves your machine is what you explicitly send to the LLM provider.

We considered optional telemetry for understanding usage patterns, but decided against it. Even anonymized telemetry can leak information about what tools you use and when.

Open Source as a Guarantee

Local-first claims are only credible when you can verify them. q is open source under the MIT license:

# Verify q's network behavior yourself
grep -r "http-request" src/
grep -r "websocket" src/
grep -r "telemetry" src/

Every network call is visible in the source code. No hidden phone-home, no analytics SDK, no third-party tracking.

What About Cloud Features?

Some features benefit from cloud infrastructure — shared sessions, team knowledge bases, model fine-tuning on your code patterns. These are possible future additions, but they will always be opt-in. The local-first experience will never require them.

The Trade-off

Local-first means you manage your own API keys and configuration. There's no dashboard, no team management portal, no central billing. For individual developers and small teams, this is a feature, not a limitation. For large organizations, we're exploring enterprise deployment options that keep code on-premises while providing centralized management.

"The best way to keep secrets is to not share them. Local-first isn't just a technical choice — it's a values choice." — q contributor
