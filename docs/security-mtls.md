# Security Model: Mutual TLS (mTLS) for Distributed Execution

This document describes the security architecture of q's Phase 2 distributed execution feature (v0.99.29). It covers the trust model, threat model, defense-in-depth layers, and key rotation procedures.

## Trust Model

### Root of Trust: Local CA

The Certificate Authority (CA) is the root of trust. The CA certificate (`ca-cert.pem`) signs all server and client certificates. Trust is established when:

1. **Both machines share the same CA certificate.** This is the out-of-band trust bootstrap.
2. **Server identity** is verified by the orchestrator: the executor's certificate must chain to the shared CA.
3. **Client identity** is verified by the executor: the orchestrator's certificate must chain to the shared CA.

This is **mutual TLS (mTLS)** — both sides authenticate each other cryptographically.

### What the CA Key Means

- Whoever holds `ca-key.pem` can issue new certificates for the cluster.
- **Protect `ca-key.pem` at all costs.** It should never be copied to executor nodes or stored in version control.
- The CA key is only needed during `q generate-certificates`. After that, it can be stored offline.

### Certificate Properties

All certificates generated by `q generate-certificates` use:
- **RSA 4096-bit keys** — strong against classical attacks.
- **365-day validity** — must be rotated annually.
- **SHA-256 signatures** — modern hash algorithm.
- **TLS 1.2** — q uses TLS 1.2 exclusively (TLS 1.3 is available but 1.2 is universally supported).

## Threat Model

### What mTLS Protects Against

| Threat | Mitigation |
|--------|------------|
| **MITM (Man-in-the-Middle)** | mTLS encrypts all traffic and authenticates both endpoints. An attacker cannot intercept or modify requests in transit. |
| **Unauthorized executor access** | The executor rejects connections that don't present a valid client certificate. |
| **Unauthorized orchestrator spoofing** | The executor verifies the client certificate chains to the trusted CA. |
| **Network sniffing** | All traffic is encrypted via TLS 1.2. |
| **Token replay on the wire** | Capability tokens have a 5-minute TTL. Replayed tokens expire quickly. |
| **Token forgery** | Tokens are HMAC-SHA256 signed. Without the shared secret, forging is computationally infeasible. |

### What mTLS Does NOT Protect Against

| Non-threat | Explanation |
|------------|-------------|
| **Compromised CA key** | If `ca-key.pem` is stolen, the attacker can mint certificates and impersonate any node. Protect the CA key. |
| **Compromised capability secret** | If the shared HMAC secret is leaked, an attacker can forge valid tokens. Protect the secret. |
| **Compromised executor node** | If the executor is rooted, the attacker can execute arbitrary code on it. This is expected — the executor is the isolation boundary. |
| **DNS spoofing** | mTLS authenticates via certificate chain, not DNS name. DNS spoofing alone won't break mTLS. |
| **Side-channel attacks** | Not mitigated by mTLS. If side-channel resistance is required, use dedicated hardware. |

### Attack Surface When Disabled

When `mas.broker.enabled = false` (the default):
- **Zero TCP listeners** are started.
- **Zero network sockets** are opened.
- **Zero certificate files** are read.
- **Zero remote executor connections** exist.

The broker code is present in the binary but completely inert. There is no attack surface when the feature is disabled.

## Defense in Depth

q's distributed execution uses four independent security layers:

```
Layer 4: EXECUTION SANDBOX
  ┌─────────────────────────────────────────────────┐
  │ Resource limits: CPU, memory, timeout, processes │
  │ Read-only repo mount · Scratch write dir        │
  │ No network egress from sandbox                   │
  └───────────────────────┬─────────────────────────┘
                          │
Layer 3: DEPLOYMENT GATE
  ┌───────────────────────┴─────────────────────────┐
  │ mas.broker.enabled = false (default)             │
  │ Capability secret required when enabled          │
  │ Cert directory required when enabled             │
  │ Fail-closed if misconfigured                     │
  └───────────────────────┬─────────────────────────┘
                          │
Layer 2: APPLICATION AUTH (Capability Tokens)
  ┌───────────────────────┴─────────────────────────┐
  │ HMAC-SHA256 signed tokens (per-request)          │
  │ TTL: 5 minutes (replay protection)               │
  │ Agent-scoped (token bound to agent-id)           │
  │ Capability-scoped (token bound to action type)   │
  │ Constant-time MAC comparison                     │
  └───────────────────────┬─────────────────────────┘
                          │
Layer 1: TRANSPORT (mTLS)
  ┌───────────────────────┴─────────────────────────┐
  │ Mutual TLS 1.2 (both sides authenticated)        │
  │ RSA 4096-bit certificates                        │
  │ Certificate chain verified to trusted CA         │
  │ No plaintext ever on the wire                    │
  └─────────────────────────────────────────────────┘
```

Each layer must be independently breached for a successful attack. Compromising any single layer does not compromise the system:

- Stolen client cert (Layer 1) → attacker still needs the capability secret (Layer 2).
- Stolen capability secret (Layer 2) → attacker still needs a valid client cert (Layer 1).
- Both certs and secret (Layers 1-2) → executor sandbox limits (Layer 4) still constrain damage.

## Capability Token Security

Capability tokens are HMAC-SHA256 authenticated strings that authorize specific actions. They are generated **per-request** by the orchestrator and validated by the executor.

### Token Lifecycle

1. **Signing**: When a high-risk tool request is routed remote, `execute-via-remote` signs a capability token with the shared HMAC secret. The token includes:
   - Capability symbol (e.g., `'execute-tools`)
   - Agent ID (the requesting agent)
   - Timestamp (for TTL validation)
   - Expiry timestamp
   - HMAC signature

2. **Transmission**: The token is included in the JSON-RPC request envelope sent over the mTLS channel.

3. **Validation**: The executor's `validate-request-capability` checks:
   - HMAC signature matches (constant-time comparison)
   - Token is not expired (within TTL window)
   - Token is not from the future (clock skew tolerance)
   - Token is not replayed (not applicable — per-request tokens)

4. **Enforcement**: If validation fails, the executor returns an error response without executing any tool.

### Security Properties

- **Empty secrets rejected**: The HMAC secret must be non-empty. Empty secrets are rejected at the contract boundary.
- **Constant-time comparison**: MAC comparison uses constant-work comparison to prevent timing attacks.
- **Strict token grammar**: Tokens must have exactly five non-empty colon-separated fields. Malformed tokens return `#f`, not an error.
- **Agent scoping**: Tokens are bound to a specific agent ID. A token issued for agent A cannot be used by agent B.

## Circuit Breaker Security

The circuit breaker is a resilience mechanism that also has security implications:

### Connection Storm Prevention

When the remote executor is down or unreachable, the circuit breaker prevents the orchestrator from generating unlimited connection attempts:

- **Threshold**: After 5 consecutive failures, the circuit opens.
- **Fast-fail**: All subsequent requests fail immediately (no connection attempt, no timeout wait).
- **Cooldown**: After 30 seconds, the circuit transitions to half-open and allows one trial request.
- **Recovery**: If the trial succeeds, the circuit closes. If it fails, the circuit re-opens.

This prevents:
- **Resource exhaustion**: No thread/connection pool drain from retry storms.
- **Amplified latency**: Requests fail in microseconds instead of timing out in seconds.
- **Information leakage**: Fast-fail responses contain no connection-level details.

## Key Rotation

### Certificate Rotation

Certificates expire after 365 days. Rotate before expiry:

1. **On the orchestrator:**
   ```bash
   q generate-certificates --force --output-dir ~/.q/certs
   ```
   This regenerates the CA and all certs. The old CA key is overwritten.

2. **Distribute new certs:**
   - Copy `ca-cert.pem`, `server-cert.pem`, `server-key.pem` to the executor.
   - Copy `ca-cert.pem`, `client-cert.pem`, `client-key.pem` to the orchestrator.

3. **Restart both nodes.** Active connections will be re-established with new certs.

> **Zero-downtime rotation:** If downtime is unacceptable, generate a second CA and certs, add the new CA to the trust store, then switch over and remove the old CA.

### Capability Secret Rotation

1. **Choose a new secret.** Use a strong random string:
   ```bash
   openssl rand -base64 32
   ```

2. **Update config on both machines simultaneously:**
   ```yaml
   mas:
     broker:
       capability-secret: "<new-secret>"
   ```

3. **Restart both nodes.** During the brief window, requests may fail with token validation errors. This is expected.

## Security Audit Checklist

Use this checklist to verify your deployment is secure:

- [ ] `mas.broker.enabled` is `false` by default on all non-executor machines.
- [ ] `ca-key.pem` is stored offline and never deployed to executor nodes.
- [ ] `capability-secret` is a strong random string (32+ characters).
- [ ] Firewall restricts port 8443 to known IPs only.
- [ ] Executor node has no network egress (sandbox enforced).
- [ ] Certificates are rotated before 365-day expiry.
- [ ] Trace logging is disabled in production (it may log sensitive routing decisions).
- [ ] The executor node runs with minimal privileges (non-root user).
- [ ] The cert directory permissions are `700` (owner read/write only).
