# Release Automation Policy — v0.99.40

## Purpose

This document defines the canonical release automation model for the
Pi-Q project. It supersedes all prior release trigger assumptions and
establishes the enforceable contract for milestone completion.

## 1. Release Trigger Model

The canonical release trigger is **tag-push** (Model A):

```text
git tag vX.Y.Z
git push origin vX.Y.Z
```

Pushing a semver tag (`v*`) triggers `.github/workflows/release.yml`,
which builds the tarball, generates release notes and manifest, creates
the GitHub Release, and runs post-release smoke tests.

### Non-Triggers

- **Milestone close does NOT create releases.** Closing a milestone is
  gated on a verified release existing (see Section 4), but does not
  itself trigger publication.
- **Branch merges do NOT create releases.** Only tagged commits are
  released.
- **Manual workflow dispatch does NOT create the canonical release.**
  The release-repair.yml workflow is for backfill/repair only.

## 2. Pre-Tag Gates

Before creating a tag, the following gates MUST pass locally:

| Gate | Script | Purpose |
|------|--------|---------|
| Version consistency | `racket scripts/lint-version.rkt` | All version references match `util/version.rkt` |
| Release readiness | `racket scripts/lint-release-readiness.rkt --strict` | No uncommitted changes, version is bumped |
| Release dry-run | `racket scripts/release-dry-run.rkt` | Validates tarball/notes/manifest generation |
| Pre-release check | `racket scripts/pre-release-check.rkt` | Combined lint-version + metrics + prose |
| Fast test suite | `racket scripts/run-tests.rkt --suite fast` | No regressions in fast tier |

### Pre-Tag Sequence

```bash
cd q/
racket scripts/lint-version.rkt
racket scripts/lint-release-readiness.rkt --strict
racket scripts/release-dry-run.rkt --context tag-publish
racket scripts/pre-release-check.rkt
racket scripts/run-tests.rkt --suite fast
```

## 3. Tag-Publish Gates

The `release.yml` workflow runs these gates after tag push:

1. **setup-racket**: Package installation must succeed (v0.99.39 fix)
2. **Version consistency**: Tag version must match `util/version.rkt`
3. **Strict release readiness**: `--strict --context tag-publish`
4. **Full test suite**: All tests pass under `STRICT_TEST_RUNNER=1`
5. **Gate evidence**: arch, workflows, tui, fast suites recorded
6. **Asset generation**: Tarball + manifest + release notes generated
7. **Asset verification**: Release exists with tarball + manifest assets

### Failure Handling

If any gate fails:
- No GitHub Release is created
- The tag remains (is not deleted)
- The failure is visible in GitHub Actions
- The release must be repaired via `release-repair.yml` after fixing

## 4. Milestone Completion Sequence

A milestone may only be closed after ALL of the following steps
complete successfully:

```text
1. All milestone issues are closed/merged.
2. Local release gates pass (Section 2).
3. Pre-tag readiness passes.
4. Tag is created and pushed.
5. Release workflow (release.yml) succeeds.
6. GitHub Release assets are verified (tarball + manifest).
7. Smoke/post-release classification is recorded.
8. milestone-gate passes (release truth check).
9. Milestone is closed (close_milestone_complete requires release).
```

### Enforcement

- `close_milestone_complete()` in gh_helpers.py requires a verified
  release by default (`require_release=True`).
- `milestone-gate.rkt` performs asset-level release truth checks.
- Assetless releases do NOT count as complete.

## 5. GitHub Release Asset Requirements

Every GitHub Release MUST have:

| Asset | Generated by | Purpose |
|-------|-------------|---------|
| `q-X.Y.Z.tar.gz` | release.yml (tar) | Source tarball for installation |
| `release-manifest.json` | gen-release-manifest.rkt | Version + file count metadata |

### Verification

Both assets are verified:
- After creation in release.yml (REST API check)
- By milestone-gate.rkt before milestone close
- By verify_milestone_release() in gh_helpers.py

A release without both assets is considered **incomplete** and blocks
milestone closure.

## 6. Smoke / Post-Release Handling

The `release.yml` workflow includes a `smoke` job that:

1. Downloads the published tarball
2. Installs it in a clean environment
3. Verifies `q --version` outputs the correct version
4. Verifies manifest version matches
5. Runs the full test suite against the tarball

### Smoke Classification

- **PASS**: All smoke checks succeed → release is verified
- **FAIL**: Any smoke check fails → release exists but is unverified
- **SKIPPED**: Smoke job didn't run → must be manually triggered

A failed smoke job does NOT delete the release, but the milestone
should not be closed until the failure is resolved or documented.

## 7. Repair / Backfill Procedure

For tags that exist but lack releases or have incomplete assets:

### Tooling

- **Workflow**: `.github/workflows/release-repair.yml`
- **Script**: `scripts/release-repair.rkt`
- **Policy**: `docs/reports/RELEASE-BACKFILL-POLICY-v0.99.40.md`

### Modes

| Mode | Behavior |
|------|----------|
| `dry-run` (default) | Verify readiness, no publication |
| `publish` | Create GitHub Release if absent |
| `repair-assets` | Upload missing assets to existing release |

### Safety Guarantees

- Default mode is dry-run (no accidental publication)
- Refuses if tag version ≠ canonical version
- Refuses if CHANGELOG entry is missing
- Never mutates git tags
- Does not claim historical gates passed unless rerun

See `RELEASE-BACKFILL-POLICY-v0.99.40.md` for tag classification tiers.

## 8. Emergency Override Policy

If automated release creation is impossible and a release MUST be
published manually:

1. **Verify readiness locally**:
   ```bash
   racket scripts/release-dry-run.rkt
   racket scripts/lint-release-readiness.rkt --strict --context tag-publish
   ```

2. **Create release manually** via GitHub UI or `gh release create`

3. **Upload assets**: tarball + manifest

4. **Verify** with `verify_milestone_release()` from gh_helpers.py

5. **Document**: Record the override reason in the milestone issue

Emergency overrides should be rare and must be reviewed post-hoc.

## 9. Required Evidence for Final Audit

For a milestone to be considered fully complete, the following evidence
must be available:

| Evidence | Source |
|----------|--------|
| Tag exists at correct commit | `git tag` |
| GitHub Release exists with 2 assets | GitHub REST API |
| release.yml workflow succeeded | GitHub Actions |
| Smoke job passed (or classified) | GitHub Actions |
| milestone-gate.rkt passed | Local run |
| Version consistency across all docs | `lint-version.rkt` |
| CHANGELOG entry exists | `CHANGELOG.md` |
| No stale board states | `verify_milestone()` |
| All sub-issues closed | GitHub Issues |

## 10. CI Dependency

The entire release automation depends on the `setup-racket` composite
action (`.github/actions/setup-racket/action.yml`) successfully
installing the package. This was the root cause of the 375/375 release
failures after v0.32.9.

### Remediation (v0.99.39)

- F1–F4 compile errors in scripts/CLI modules fixed
- `ci-package-setup.rkt` provides local CI-equivalent gate
- `--package-setup` flag on `ci-local.rkt` for pre-PR validation
- Setup-racket action has explicit naming and diagnostics

### Verification

Every CI run is checked for setup-racket success. The step named
`Run ./.github/actions/setup-racket` must show `conclusion=success`.
